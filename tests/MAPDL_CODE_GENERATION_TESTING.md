# MAPDL Code Generation Testing Framework

This framework enables testing of MAPDL code generated by an LLM when using the `PYMAPDL_SYSTEM_PROMPT` from the pymapdl-mcp MCP server.

## Overview

The testing framework follows a pattern similar to [aali-codegen](https://github.com/ansys-internal/aali-codegen):

1. **Query Generation**: Send a natural language query to an LLM (Azure OpenAI) along with the `PYMAPDL_SYSTEM_PROMPT`
2. **Code Extraction**: Extract executable Python code from the LLM response
3. **Code Adaptation**: Clean up the generated code for safe test execution
4. **Code Execution**: Run the adapted code against a MAPDL instance (real or mock)
5. **Result Validation**: Verify that the code produced expected outcomes

## Architecture

### Core Components

- **`test_utils.py`**: Utility functions for code extraction and adaptation
  - `extract_executable_code()`: Extracts Python code from various LLM response formats
  - `adapt_code_for_execution()`: Removes problematic imports/commands for test context
  - `get_llm_context()`: Returns the system prompt and tool descriptions

- **`test_mapdl_code_generation.py`**: Base template class for building tests
  - `TemplateTest`: Reusable base class with common test patterns
  - Built-in validation methods for entities, commands, parameters, etc.

- **`test_mapdl_code_generation_examples.py`**: Example test classes
  - `TestSimpleKeypoint`: Example of testing simple geometry creation
  - `TestTwoKeypoints`: Example of testing multiple entities and relationships

## Writing Tests

### Basic Test Structure

```python
from test_mapdl_code_generation import TemplateTest
import pytest

@pytest.mark.llm_integration
class TestYourMAPDLTask(TemplateTest):
    """Test that the LLM can generate code for your task."""

    # 1. Define what you want the LLM to do
    query = "Your task description here"

    # 2. Specify expected outcomes
    num_keypoints = 2
    num_lines = 1
    keypoints = [1, 2]

    # 3. Define what to validate in generated code
    commands = ["k", "l"]  # PyMAPDL methods that should be called

    # 4. Set up test execution
    @pytest.fixture(autouse=True)
    def setup_and_execute(self, mapdl_fixture, llm_client):
        self.setup_class()
        mapdl_context = get_llm_context()
        self.run_llm_query(llm_client, mapdl_context)
        self.extract_code()
        self.adapt_code()
        self.execute_code(mapdl_fixture)
        yield mapdl_fixture
```

### Test Configuration Attributes

Set these class attributes to define what your test expects:

#### Expected Entity Counts
```python
num_nodes = 100          # Expected number of nodes
num_elements = 50        # Expected number of elements
num_keypoints = 4        # Expected number of keypoints
num_lines = 3            # Expected number of lines
num_areas = 2            # Expected number of areas
num_volumes = 1          # Expected number of volumes
```

#### Specific Entity IDs
```python
keypoints = [1, 2, 3]    # Expected keypoint IDs (if important)
lines = [1, 2]           # Expected line IDs
areas = [1]              # Expected area IDs
```

#### Code Validation
```python
commands = ["k", "l", "a"]   # PyMAPDL methods expected in code
words = ["keypoint", "line"]  # Words expected in LLM response
parameters = {"SIZE": 0.1}   # Parameter values expected in model
```

#### Custom Validation
```python
def validate_geometry(mapdl):
    """Custom validation logic."""
    # Access MAPDL instance to verify details
    kp1 = mapdl.get_value("KP", 1, "LOC", "X")
    assert abs(kp1 - 0.0) < 1e-6, "Keypoint 1 X-coordinate incorrect"

functions = [validate_geometry]  # List of validation functions
```

### Built-in Validation Methods

The `TemplateTest` class provides automatic test methods:

- **`test_code_execution_succeeds`**: Verifies generated code runs without errors
- **`test_entity_count`**: Checks that the correct number of each entity type was created
- **`test_command_usage`**: Verifies expected PyMAPDL methods are in the generated code
- **`test_response_contains_words`**: Checks for specific words in LLM response
- **`test_parameters`**: Validates MAPDL parameter values
- **`test_custom_functions`**: Runs custom validation functions on the MAPDL instance

## Running Tests

### Prerequisites

Install test dependencies:
```bash
pip install -e ".[tests]"
```

Or manually install:
```bash
pip install pytest openai genai-test pytest-vcr
```

### Environment Variables

For LLM tests, set Azure OpenAI credentials:
```bash
export AZURE_OPENAI_KEY="your-api-key"
export AZURE_OPENAI_ENDPOINT="https://your-deployment.openai.azure.com/"
export AZURE_OPENAI_DEPLOYMENT="your-deployment-name"
```

To use a real MAPDL instance instead of mock:
```bash
export USE_REAL_MAPDL="true"
```

### Running Tests

Run all LLM integration tests:
```bash
pytest tests/test_mapdl_code_generation_examples.py -v -m llm_integration
```

Run only unit tests (no LLM):
```bash
pytest tests/test_test_utils.py -v -m unit
```

Run a specific test class:
```bash
pytest tests/test_mapdl_code_generation_examples.py::TestSimpleKeypoint -v
```

Run tests with verbose output:
```bash
pytest tests/test_mapdl_code_generation_examples.py -xvs
```

## How It Works

### 1. Code Extraction

The `extract_executable_code()` function handles various LLM response formats:

- **Markdown code blocks**: ` ```python ... ``` `
- **Plain code blocks**: ` ``` ... ``` `
- **Code patterns**: Automatically detects lines starting with Python keywords

```python
response = """Here's the code:
```python
from ansys.mapdl.core import launch_mapdl
mapdl = launch_mapdl()
mapdl.k(1, 0, 0, 0)
```
"""

code = extract_executable_code(response)
# Returns: "from ansys.mapdl.core import launch_mapdl\nmapdl = launch_mapdl()..."
```

### 2. Code Adaptation

The `adapt_code_for_execution()` function prepares code for test execution by commenting out:

- MAPDL launch/initialization (we provide our own instance)
- MAPDL exit commands
- Ansys package imports (we handle these)

```python
# Original code
code = """
from ansys.mapdl.core import launch_mapdl
mapdl = launch_mapdl()
mapdl.prep7()
mapdl.k(1, 0, 0, 0)
mapdl.exit()
"""

adapted = adapt_code_for_execution(code)
# Resulting code has imports/launch/exit commented out
# But keeps the actual MAPDL operations: k(), prep7()
```

### 3. Stateful Mock MAPDL

When not using a real MAPDL instance, tests use a stateful mock that:

- Tracks entity creation (`k()`, `l()`, `a()`, `v()`)
- Updates entity counts dynamically
- Records parameters
- Executes commands without errors

This allows tests to validate that code correctly creates geometry without requiring a MAPDL license.

## Best Practices

### 1. Keep Queries Deterministic

Choose queries with reliable, reproducible outputs:
- ✅ "Create a keypoint at (0,0,0)" → always one keypoint at specific location
- ❌ "Create a random mesh" → non-deterministic output

### 2. Start Simple

Begin with basic geometry operations:
- Single keypoint creation
- Multiple keypoints and lines
- Simple areas and boxes

Then expand to more complex operations.

### 3. Use Meaningful Query Descriptions

```python
# Good: Clear, specific about expected result
query = "Create a 10mm line between keypoints at (0,0,0) and (10,0,0)"

# Bad: Vague, multiple interpretations
query = "Create a line"
```

### 4. Validate Incrementally

```python
# Good: Check multiple aspects
num_keypoints = 2
num_lines = 1
commands = ["k", "l"]
words = ["keypoint", "line"]

# Less useful: Only check one thing
num_keypoints = 2
```

### 5. Document Custom Validators

```python
def check_line_length(mapdl):
    """Verify line has correct length."""
    # Include clear docstrings explaining what's validated
    length = mapdl.get_value("LINE", 1, "LENGTH")
    assert abs(length - 10.0) < 0.1, "Line length incorrect"

functions = [check_line_length]
```

## Troubleshooting

### Code Extraction Fails

If you see "Could not extract valid Python code":

1. Check that the LLM response contains Python code
2. Try wrapping code in markdown: ` ```python ... ``` `
3. Check for syntax errors in the response using a Python AST checker

### Execution Fails

If "Code execution failed":

1. Check the adapted code - look for commented imports that broke logic
2. Verify the MAPDL instance is properly initialized
3. Check if the code tries to use methods not in the mock

### Entity Counts Don't Match

If entity counts are wrong:

1. Verify the LLM actually created the expected entities (check `commands`)
2. Check if mock needs updates for new MAPDL methods
3. Use custom validators to debug the actual state

### LLM Test Skipped

If tests are skipped:

1. Check that Azure OpenAI credentials are set
2. Verify the openai package is installed
3. Run with `-xvs` flag to see the skip reason

## Response Caching

By default, LLM responses are cached using pytest-vcr. This:

- Reduces costs (no repeated API calls)
- Speeds up tests (uses cached responses)
- Ensures reproducibility

Caches are stored in `tests/cassettes/`. To invalidate cache:

```bash
rm -rf tests/cassettes/  # Force fresh LLM queries
```

## Architecture Decisions

### Why Stateful Mock MAPDL?

The framework uses a stateful mock MAPDL that tracks entity creation because:

1. **CI/CD Compatibility**: Tests can run without MAPDL license
2. **Speed**: Mock operations are instant, no solver overhead
3. **Determinism**: No randomness from solver, purely validation of code logic
4. **Debugging**: Easy to inspect state without complex MAPDL queries

### Why Comment Out Imports Instead of Removing?

When adapting code, imports are commented rather than removed because:

1. **Debugging**: Comments show what was removed (helpful for diagnosis)
2. **Line Numbers**: Preserved for error reporting
3. **Code Structure**: Easier to reason about what changed

### Why pytest-vcr for LLM Responses?

Recording/replaying LLM responses provides:

1. **Cost Control**: First run pays, subsequent runs are free
2. **Determinism**: Same response every time (no LLM variation)
3. **CI/CD Friendly**: Tests work without credentials once cached
4. **Network Independence**: Tests don't depend on API availability
